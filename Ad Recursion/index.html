The Real Problem

You're thinking about execution. You're tracing through code line-by-line, following the call stack in your head. Strong recursive thinkers don't do this. They think about the state-space — a tree of possibilities — and write code that explores it.

This guide rewires that mental model. Every section reinforces the same four concepts: STATE · DECISION · TRANSITION · BASE CASE.

02

The 4 Foundations

Every recursion problem, without exception, reduces to four questions: What is my state? What are my decisions? How does state change? When do I stop?

① STATE

Definition

State is the complete snapshot of information at a single recursion node. It is everything a function call needs to make its decisions — without knowing anything about parent calls or future calls.

Why it matters: State directly determines your function's parameters. If you don't identify state correctly, your function will either carry too little information (wrong answers) or too much (over-complicated logic).

The test: If I give you only the state at a node, can you independently decide what to do next? If yes, the state is correct. If you need to know what the parent did, your state is incomplete.

State Examples Across Problems

ProblemState VariablesMeaning
Subsets([1,2,3]) idx=1, current=[1] Processed index 0 (included 1), now deciding about index 1 onwards
Combination Sum([2,3,6,7], 7) idx=0, rem=5, current=[2] At index 0 again, need 5 more, have [2] so far
N-Queens (N=4) row=2, cols={0,2}, diag1={-2,1} Placing row 2, columns 0 and 2 already used in previous rows
Sudoku Solver (r=1, c=3, board=partial) Filling cell (1,3), board has some digits placed
Beginner Mistake

Treating idx as a loop counter. The index in recursion is part of the state — it represents "which choices are still available to me." This is a fundamentally different mental model.

Interviewer POV

When you say "Let me think about what my function needs to know at each call" before writing any code, interviewers recognize this as senior-level thinking. State identification IS the algorithm.

② DECISION

Definition

A decision is the set of choices available from the current state. Each choice spawns a new branch in the recursion tree — a child node.

Why it matters: The decisions determine the shape of your recursion tree. Two decisions = binary tree. N decisions = N-ary tree. Identifying decisions precisely tells you how many recursive calls you make and what loop structure (if any) you need.

Decision Examples

ProblemDecisions at Each NodeTree Branching
Subsets Include nums[idx] OR exclude nums[idx] Binary (2 branches)
Combination Sum I Skip current element OR take it (possibly again) Binary (but with index behavior difference)
Combination Sum II For each position i from start to end, try nums[i] Loop-based (up to N branches)
N-Queens Place queen in column 0, 1, 2, ..., N-1 of current row N branches (filtered by constraints)
Sudoku Place digit 1–9 in the current empty cell Up to 9 branches (filtered by constraints)
Beginner Mistake

Difficulty identifying decisions usually comes from thinking "what does the code do?" instead of "what choices does this problem offer?" Step back. Ask: at this moment in the problem, what are my options? List them. Those are your decisions.

③ TRANSITION

Definition

A transition describes how state changes when you make a particular decision. It defines what arguments you pass to the next recursive call — and what you undo when backtracking.

Why it matters: Transitions ARE your recursive calls. Once you know "if I make decision X, state becomes Y," writing the recursive call is mechanical. The undo of a transition is backtracking.

Transition Examples

ProblemDecisionNew State (Transition)Undo (Backtrack)
Subsets Include nums[idx] idx+1, current+[nums[idx]] current.pop()
Subsets Exclude nums[idx] idx+1, current unchanged Nothing to undo
Combination Sum I Take nums[idx] same idx, rem - nums[idx], current+[nums[idx]] current.pop()
N-Queens Place queen at (row, col) row+1, cols+{col}, diag1+{row-col}, diag2+{row+col} Remove from all sets
Sudoku Place digit d at (r,c) next empty cell, board[r][c]=d board[r][c]=0
Beginner Mistake

Confusing validation with state modification. Checking "is this decision valid?" is not a transition — it's a guard before the transition. Modifying the state IS the transition. These two things belong in separate places in your code.

④ BASE CASE

Definition

The base case is the condition that stops recursion. There are two types: a success base case (record a result) and a failure base case (prune / return without recording).

Why it matters: Without a correct base case, recursion runs forever or misses results. Your base case defines what "done" looks like for your problem.

Base Case Examples

ProblemSuccess ConditionFailure Condition
Subsets idx == len(nums) → append current to results N/A (every path is valid)
Combination Sum I rem == 0 → append current to results rem < 0 or idx >= len(nums)
N-Queens row == n → append board layout Constraint violation before recursive call (pruned)
Sudoku No empty cells remain → return True No digit 1-9 is valid for current cell → return False
The Master Formula

Every recursion / backtracking problem = define STATE → enumerate DECISIONS → compute TRANSITION for each → check BASE CASE at start of function. Write in this order before touching code.

Full Example: Subsets Decomposed

State

(idx, current_subset)
idx = which element we're deciding about now. current_subset = elements chosen so far.

Decision

Two choices: include nums[idx] in current_subset, OR exclude it.

Transition

Include: idx+1, current + [nums[idx]]
Exclude: idx+1, current unchanged

Base Case

idx == len(nums) → every element decided. Record current_subset. Return.

03

Recursion Tree Thinking

Recursion is not code execution. Recursion is state-space exploration. Your recursive function does not "run" — it navigates a tree of states.

The Mental Model Shift

Weak Thinker (Code Execution)

"When I call backtrack(1, [1]), then inside that function it calls backtrack(2, [1,2]) and also backtrack(2, [1])..."

This person is mentally executing code. They get lost after 2-3 levels of recursion.

Strong Thinker (State-Space)

"The tree has nodes representing states. From state (idx=1, [1]), I have two children: (idx=2, [1,2]) and (idx=2, [1]). All paths from root to leaf are my answers."

This person sees the shape. They can reason about the problem without tracing.

The Three Rules of Recursion Trees

Rule 1: Nodes = States

Each function call is a node. The node label is the complete state at that call. You should be able to read the node and know exactly where you are in the problem.

Rule 2: Edges = Decisions

Edges connect parent states to child states. Each edge is labeled with the decision made. The number of edges from a node = number of choices at that state.

Rule 3: Paths from Root to Leaf = Exploration History

Each path from root to any leaf node tells you the sequence of decisions made. Leaf nodes that satisfy the success base case are your answers.

The Key Insight

Notice that in all three trees, you never think about code when drawing these trees. You think about states and choices. Once the tree is correct, the code writes itself: each node = one function call, each edge = one recursive call, leaf = base case.

Recursion Pattern Classification

# Pattern Name Decision Type State Transition Key Problems
1 Linear Recursion One recursive call n n → n-1 Factorial, Linked List
2 Binary Recursion Split into two halves (lo, hi) → (lo,mid), (mid+1,hi) Merge Sort, Binary Search
3 Take / Not Take Include or exclude current element (idx, current) idx+1, current±elem Subsets, Subsets II, 0-1 Knapsack
4 Multi-Choice (Reusable) Take current (allow repeat) or skip (idx, rem, current) Take: idx stays, rem-val; Skip: idx+1 Combination Sum I
5 Loop-Based Backtracking Loop from start, pick each valid element (start, current, rem) start=i+1, current+[nums[i]], rem-nums[i] Combination Sum II, III, Combinations
6 Permutation Pattern Loop through all unused elements (used[], current) Mark used, add to current; undo both Permutations I, II
7 Combination Pattern Pick elements in sorted order (no reuse) (start, k, current) start=i+1, k stays, current+[i] C(n,k), Combination Sum III
8 Constraint Backtracking Place value; validate against constraints first (position, constraint_sets) Place at position + update constraints; undo both N-Queens, Sudoku, Generate Parentheses
9 Branch and Bound Same as constraint backtracking + bound check (pos, current_cost, bound) Prune if current exceeds known best TSP, Graph coloring
10 DP Transition Pattern Same as recursion + memoize overlapping states (idx, rem) + memo Same transitions + memo[state] = result Coin Change, Unique Paths, House Robber
Pattern Selection Rule

Start with the decision type. Binary decision (include/exclude) → Take/Not Take. Loop over remaining choices → Loop-Based. Need to handle constraints → Constraint Backtracking. Overlapping subproblems → DP Transition. The table above maps naturally.

05

Complete Backtracking Pattern Roadmap

P1 Take / Don't Take Binary · 2^n tree · No loop

Recognition clue: Problem asks for all subsets, subsequences, or combinations where each element appears at most once and you decide per element independently.

State

(idx, current) — index into the array, elements chosen so far

Decision

2 choices: include nums[idx] OR exclude it

Transition

Include: current.append(nums[idx]), recurse with idx+1
Exclude: recurse with idx+1, current unchanged

Base Case

idx == len(nums) → record current (snapshot it!)

Problems: Subsets Subsets II (sort + skip dups) Subsequence with sum K

Common mistake: For Subsets II (with duplicates), you must sort the array first, then in the "exclude" path, skip all elements equal to nums[idx] before recursing — not after.

# SUBSETS — Take / Don't Take
def subsets(nums):
    results = []
    def bt(idx, current):
        if idx == len(nums):          # BASE CASE: all elements decided
            results.append(current[:])  # snapshot
            return

        current.append(nums[idx])   # TAKE: modify state
        bt(idx + 1, current)        # TRANSITION: advance
        current.pop()               # UNDO: backtrack

        bt(idx + 1, current)        # DON'T TAKE: advance without adding

    bt(0, [])
    return results
P2 Reusable Choice Binary · Infinite repetition allowed

Recognition clue: "You may use the same element multiple times." The key twist: when you "take" an element, the index does NOT advance.

State

(idx, rem, current) — index, remaining sum, current combination

Decision

Skip current element (advance idx) OR take it again (stay at idx, reduce rem)

Transition

Skip: idx+1, rem unchanged
Take: idx unchanged, rem - nums[idx]

Base Case

rem == 0 → success; rem < 0 or idx >= len → fail

Problems: Combination Sum I

Pruning insight: If you sort the array first, you can break when nums[idx] > rem — all subsequent elements are also too large.

# COMBINATION SUM I — Reusable Choice
def combinationSum(candidates, target):
    candidates.sort()  # enable pruning
    results = []
    def bt(idx, rem, current):
        if rem == 0:                       # BASE CASE: found valid combo
            results.append(current[:])
            return
        if idx >= len(candidates): return  # BASE CASE: exhausted choices

        for i in range(idx, len(candidates)):
            if candidates[i] > rem: break      # PRUNE: sorted, no point continuing
            current.append(candidates[i])
            bt(i, rem - candidates[i], current) # i not i+1 = allow reuse
            current.pop()

    bt(0, target, [])
    return results
P3 Loop-Based Combinations N-ary · start index · no reuse

Recognition clue: Combinations from a set with no repetition, where order doesn't matter. You always pass a start index to avoid revisiting earlier elements.

State

(start, current) — starting index for this call's loop, elements chosen

Decision

Loop: for each i from start to end, choose nums[i]

Transition

Add nums[i], recurse with start = i+1 (no re-picking same element)

Base Case

Length/sum target met → record; or loop simply exhausts naturally

Problems: Combination Sum II Combination Sum III Combinations

Deduplication (Sum II): Sort array. In the loop, if i > start and nums[i] == nums[i-1], skip. This prevents same-value elements at the same position in the tree.

# COMBINATION SUM II — with duplicate handling
def combinationSum2(candidates, target):
    candidates.sort()
    results = []
    def bt(start, rem, current):
        if rem == 0:
            results.append(current[:])
            return
        for i in range(start, len(candidates)):
            if candidates[i] > rem: break
            if i > start and candidates[i] == candidates[i-1]: continue  # skip dup
            current.append(candidates[i])
            bt(i + 1, rem - candidates[i], current)
            current.pop()
    bt(0, target, [])
    return results
P4 Permutations N! · used[] array · all orderings

Recognition clue: Order matters. Every arrangement of elements is a valid answer. Unlike combinations, you can pick any unused element at each step — no start index.

State

(current, used[]) — permutation built so far, boolean array of which elements are used

Decision

For each index i: if not used[i], pick nums[i]

Transition

Mark used[i]=True, append nums[i]; undo both after recursion

Base Case

len(current) == len(nums) → record current permutation

Problems: Permutations I Permutations II

Deduplication (Perms II): Sort array. Skip if nums[i] == nums[i-1] and not used[i-1] — this prunes duplicate subtrees.

# PERMUTATIONS I
def permute(nums):
    results = []
    used = [False] * len(nums)
    def bt(current):
        if len(current) == len(nums):
            results.append(current[:])
            return
        for i in range(len(nums)):
            if used[i]: continue
            used[i] = True
            current.append(nums[i])
            bt(current)
            current.pop()
            used[i] = False
    bt([])
    return results
P5 Constraint Backtracking Validate before recurse · constraint sets

Recognition clue: The problem has an explicit validity rule that must hold at every step (queens can't attack each other; parentheses must be balanced; Sudoku rules). You check the constraint BEFORE making the recursive call.

State

Current position + constraint tracking structure (sets, counts, or board)

Decision

All choices at current position; filter by constraint validity before recursing

Transition

Place value → update constraint structure → recurse → undo both

Base Case

All positions filled successfully → record result

Problems: Generate Parentheses N-Queens Sudoku Solver

# GENERATE PARENTHESES
def generateParenthesis(n):
    results = []
    def bt(s, open_count, close_count):
        if len(s) == 2 * n:
            results.append(s)
            return
        if open_count < n:              # DECISION: can add '('
            bt(s + '(', open_count+1, close_count)
        if close_count < open_count:    # DECISION: can add ')' (constraint!)
            bt(s + ')', open_count, close_count+1)
    bt('', 0, 0)
    return results
P6 String Construction Character-level decisions · partitioning

Recognition clue: Building or partitioning strings where each character or substring has a binary/multiple choice (capitalize or not, cut here or not).

State

(idx, current_path) — character index, partitions chosen so far

Decision

Palindrome Partitioning: for each end j ≥ idx, check if s[idx:j+1] is palindrome → cut there

Transition

Add substring, advance to j+1; pop and try next j

Base Case

idx == len(s) → all characters partitioned

Problems: Letter Case Permutation Palindrome Partitioning

P7 Grid Backtracking 2D state · visited[][] · 4/8 directions

Recognition clue: Problem involves paths, searches, or placements on a 2D grid. State includes current cell position and visited status.

State

(row, col, idx_in_word, visited[][])

Decision

Move in 4 directions (up, down, left, right) from current cell — if in bounds, unvisited, matching

Transition

Mark cell visited, recurse to neighbor; unmark on return

Base Case

All word characters matched → return True (or record path)

Problems: Word Search Rat in a Maze

# WORD SEARCH — Grid Backtracking
def exist(board, word):
    rows, cols = len(board), len(board[0])
    visited = [[False]*cols for _ in range(rows)]

    def bt(r, c, idx):
        if idx == len(word): return True   # BASE: all chars matched
        if not (0 <= r < rows and 0 <= c < cols): return False
        if visited[r][c] or board[r][c] != word[idx]: return False

        visited[r][c] = True
        for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
            if bt(r+dr, c+dc, idx+1): return True
        visited[r][c] = False              # UNDO
        return False

    return any(bt(r, c, 0) for r in range(rows) for c in range(cols))
06

Your Personal Mistakes

These are the exact thinking failures that cause wrong code. Each one is diagnosed precisely so you can catch yourself in real time.

M1 Thinking in Code Execution Instead of State

Why it happens: Your brain has been trained on imperative programming. When you see a recursive call, your instinct is to mentally "jump into" the function and trace execution line by line.

Symptoms in code: You can't reason about recursion beyond 2-3 levels. You write recursion without drawing a tree first. You describe your solution as "first we do X, then we do Y" instead of "at any state (idx, current), we make these decisions."

Example: Asked how Subsets works, you say "it calls bt(0,[]), then inside that it appends 1 and calls bt(1,[1])..." You can't describe what happens without simulating.

Fix

Before writing any code, draw the recursion tree with state labels on each node. Never start with "it calls..." — start with "the state at any node is (idx, current)."

Mental Model: You are a mapmaker, not a traveler. A mapmaker sees all paths simultaneously. A traveler follows one path at a time. Be the mapmaker.
M2 Treating Recursion as Linear Flow

Why it happens: Linear reading habit. You think "this function runs top to bottom, then calls itself." But recursion creates a tree, not a sequence.

Symptoms: You get confused by the "position" after a recursive call returns. You don't understand why code after the recursive call (the undo/backtrack step) is crucial.

Example: You write current.append(x) and bt(idx+1), but forget current.pop() — because you're thinking "the function is done, we move on."

Fix

Think of recursion as going deep and coming back. Everything after the recursive call happens on the way UP the tree. The append-recurse-pop trio is: "apply decision → explore all futures → undo decision to try next option."

Mental Model: You're exploring a cave with a flashlight. You go in (append), explore all rooms (recurse), then walk back to the entrance (pop) to try the next tunnel.
M3 Mixing Recursion Styles Within One Solution

Why it happens: You half-understand two different patterns and combine them unconsciously — e.g., using a loop AND a binary take/not-take structure for the same problem.

Symptoms: Duplicate results. Wrong number of results. Same combination appearing multiple times with elements in different orders.

Example: Using a loop-based structure for Combination Sum I (when you should use take/skip binary), causing the same element to be picked from different loop positions.

Fix

Identify the pattern first (Section 7 decision tree). Commit to one pattern. Don't mix binary take/not-take with loops in the same recursive function — they make incompatible assumptions about the index.

Mental Model: Pick your weapon before the fight. A sword fighter doesn't switch to a bow mid-combat. Choose your recursion pattern, then execute it cleanly.
M4 Not Trusting Recursion (Over-Controlling)

Why it happens: You don't believe the recursive call will "do the right thing," so you try to manually manage what it does — pre-computing future states, adding extra logic, or adding redundant checks.

Symptoms: Functions that are much longer than they need to be. Variables passed into recursion that aren't actually part of the state. Nested if-else where simple recursion would work.

Fix

The recursive leap of faith: assume the recursive call handles its sub-problem perfectly. Your only job is to define the current node's state, make one decision, and hand off to recursion. Trust it.

Mental Model: A manager doesn't do their employees' jobs. You set the direction (decision + transition), then delegate (recursive call). The subtree handles itself.
M5 Trying to Manually Build Future States

Why it happens: You try to "look ahead" — you want to know what the combination will look like at the end before committing to a path. This is the opposite of how recursion works.

Symptoms: Pre-building result lists before the recursive call. Checking conditions that belong in the base case or the recursive step.

Fix

Recursion explores forward one step at a time. Define only the current step's transition. The final state assembles itself through repeated transitions. You never need to pre-compute the end state.

Mental Model: You don't know your destination when you start walking. You just know which direction to walk next. The destination reveals itself when you arrive (base case).
M6 Treating Index as a Loop Counter

Why it happens: In iterative code, indices count. In recursion, they mean something deeper: the index defines the boundary of remaining choices.

Symptoms: Confusion about whether to pass i or i+1 in recursive calls. Incorrect handling of "same element multiple times" vs "no repetition" problems.

The rule:
Pass i → allow reusing element at position i (Combination Sum I)
Pass i+1 → element i is done, move to next (Combination Sum II, Subsets)
No start index → permutation (any unused element is valid)

Fix

Ask yourself: "After choosing nums[i], should I be allowed to choose it again?" If yes, pass i. If no, pass i+1. This determines the semantics of every index in backtracking.

Mental Model: The index is not a counter. It is a pointer that says "all elements before me have been decided. I and everything after me are still available."
M7 Weak Pruning Intuition

Why it happens: Pruning requires reasoning about the structure of the search space. Beginners focus on making recursion correct first and never develop the habit of cutting bad branches early.

Symptoms: Code that produces correct results but TLEs. Not recognizing that a sorted array enables early termination. Not understanding that validity checks can be moved before the recursive call.

Fix

After writing correct code, always ask: "Is there any state I'm exploring that I already know will fail?" If yes, add a check at the top of the function before the decision loop. See Section 8 for full pruning methodology.

Mental Model: Pruning is not an optimization. It is early pattern recognition. A chess player doesn't consider every possible move — they immediately discard moves that are "obviously bad." Develop this intuition.
M8 Difficulty Identifying Decisions

Why it happens: You think about decisions in terms of code structure instead of the problem domain. You can't separate "what are the choices" from "how do I implement the choices."

The method: Freeze time at one recursion node. Ignore the code. Ask: in the real problem, at this moment, what can I do? List the choices in English. That list IS your decisions.

Fix

Every backtracking problem has a "what do I choose?" question. For Subsets: "do I include this element?" For N-Queens: "which column should the queen go in this row?" Articulate this in English before coding.

M9 Difficulty Identifying State Transitions

Why it happens: Transitions require you to think about how a decision changes the state — what's different after the decision vs. before it.

The method: For each decision, ask: "What in my state changes because of this decision?" That change = the transition. Everything you change on the way in must be un-changed on the way out (backtrack).

Fix

List state variables. For each decision, mark which variables change. Those are your transitions. Every changed variable needs an "undo" line below the recursive call. Check: does your undo perfectly reverse your modification?

M10 Confusing Validation and State Modification

Why it happens: Both validation (is this choice valid?) and state modification (apply this choice) happen around the recursive call, so they get tangled.

The rule:
Validate → check before modifying. If invalid, continue/break. Do NOT modify state.
Modify → change state (apply transition). Always followed by recursive call, then undo.
Never validate inside the undo section. Never modify state inside the validation section.

Fix

Structure your loop body as: if not valid: continue → modify state → recurse → undo state. This exact order, every time.

Mental Model: Validate is the gatekeeper. Modify + Recurse + Undo is the actual action. The gatekeeper never enters the room — they just decide who does.

How to Identify the Correct Pattern

When you see a new problem, run it through this decision sequence before writing any code. The questions are ordered from most to least discriminating.

Question 1
Is there a 2D grid involved? (positions on a board, matrix, or maze)
YES → Grid Backtracking (Pattern 7). State = (r, c, visited). 4-directional moves.
NO → Continue to Q2
↓
Question 2
Does the problem have an explicit validity constraint that must hold at every step? (queens can't attack, parentheses must be balanced, Sudoku rules)
YES → Constraint Backtracking (Pattern 5). Validate before each recursive call.
NO → Continue to Q3
↓
Question 3
Does ORDER matter? Is [1,2,3] a different answer from [3,2,1]?
YES → Permutation Pattern (Pattern 4). Use used[] array, no start index. Check if duplicates exist (sort + skip rule).
NO → Continue to Q4
↓
Question 4
Can the same element be reused multiple times in the same result?
YES → Reusable Choice (Pattern 2). Pass i (not i+1) in recursive call. Binary take/skip structure.
NO → Continue to Q5
↓
Question 5
Are there duplicate elements in the input that should NOT produce duplicate results?
YES → Loop-Based with dedup (Pattern 3). Sort + skip if nums[i] == nums[i-1] and i > start.
NO → Continue to Q6
↓
Question 6
Is the decision binary (include/exclude per element) or loop-based (pick from remaining elements)?
BINARY → Take/Not Take (Pattern 1). Two recursive calls per node. Think per-element decisions.
LOOP → Loop-Based Combinations (Pattern 3). One loop with start index. Think "pick next element."

Pruning

Pruning is not an optimization trick. It is recognizing that certain branches of the state space are provably dead — and refusing to enter them.

Why Pruning Matters

Without Pruning

Combination Sum on a list of 20 elements exploring all combinations = potentially millions of recursive calls. TLE guaranteed.

With Pruning

Once remaining sum goes negative, you skip an entire subtree instantly. Sorting enables this check. The difference can be 10x to 1000x.

How to Discover Pruning

After writing correct code, ask these three questions:

  1. Impossibility: Is there a state from which success is impossible? (e.g., remaining sum < 0, more queens needed than columns available). Prune these immediately.
  2. Redundancy: Will this branch produce results already covered by a previous branch? (e.g., same combination with different starting position). Use sorting + dedup to prune.
  3. Optimality: Is this branch provably worse than the best solution found so far? (Branch and Bound). Prune if current partial solution already exceeds the known optimum.

Pruning Examples

Combination Sum — Impossibility Pruning

Sort the candidates. In the loop, if candidates[i] > remaining, break. Since the array is sorted, all subsequent elements are also too large. Entire remaining loop is skipped.

for i in range(start, len(candidates)):
    if candidates[i] > rem: break   # sorted → all remaining also too large
    ...
Combination Sum III — Dual Pruning

Find k numbers that sum to n, using digits 1–9 exactly once. Two prune conditions:

  • If current sum exceeds target: break (digits are sorted)
  • If we've picked more than k numbers: return immediately
def bt(start, current, remaining):
    if len(current) > k: return        # PRUNE: too many elements
    if len(current) == k and remaining == 0:
        results.append(current[:])
        return
    for i in range(start, 10):
        if i > remaining: break            # PRUNE: impossible to reach target
        current.append(i)
        bt(i + 1, current, remaining - i)
        current.pop()
N-Queens — Constraint Pruning

Before placing a queen, check if the column and both diagonals are free. Checking takes O(1) with sets. This prunes invalid placements before exploring their subtrees — often eliminating 90%+ of the search space.

for col in range(n):
    if col in cols or (row-col) in diag1 or (row+col) in diag2:
        continue  # PRUNE: queen would be attacked
    cols.add(col); diag1.add(row-col); diag2.add(row+col)
    solve(row + 1)
    cols.remove(col); diag1.remove(row-col); diag2.remove(row+col)

Safe vs Unsafe Pruning

Safe PruningUnsafe Pruning (Bug Risk)
Prune based on provable impossibility (sum exceeds target, constraint violated) Pruning based on intuition without proof ("this looks like it won't work")
Pruning sorted arrays when current element exceeds remaining budget Pruning based on partial state without considering all remaining choices
Deduplication pruning: skip same value at same depth after sorting Pruning that causes valid results to be skipped (off-by-one in dedup condition)
Constraint violation pruning (N-Queens, Sudoku) Assuming "if first choice fails, all choices fail" (without actually checking)

Master Template Library

These templates encode the thinking framework into reusable code structures. Study the annotations — they map code to STATE / DECISION / TRANSITION / BASE CASE.

1 — Binary Recursion Split in half · merge results 
def solve(lo, hi):
    # BASE CASE
    if lo >= hi: return base_value

    mid = (lo + hi) // 2

    # DECISION: split into two subproblems
    left  = solve(lo, mid)          # TRANSITION: (lo,hi) → (lo,mid)
    right = solve(mid+1, hi)       # TRANSITION: (lo,hi) → (mid+1,hi)

    return merge(left, right)
2 — Take / Not Take Binary · 2^n tree · Subsets family 
def backtrack(idx, current):
    # BASE CASE: all elements decided
    if idx == len(nums):
        results.append(current[:])   # SNAPSHOT is critical
        return

    # DECISION 1: TAKE nums[idx]
    current.append(nums[idx])      # modify state
    backtrack(idx + 1, current)    # transition: advance
    current.pop()                  # undo

    # DECISION 2: SKIP nums[idx]
    backtrack(idx + 1, current)    # transition: advance (no undo needed)
3 — Combination Pattern (No Reuse) Loop · start index · sorted 
def backtrack(start, current):
    # BASE CASE: target achieved
    if target_achieved(current):
        results.append(current[:])
        return

    # DECISION: choose one element from [start, end)
    for i in range(start, len(nums)):
        if nums[i] > remaining: break  # PRUNE if sorted
        if i > start and nums[i] == nums[i-1]: continue  # DEDUP if needed

        current.append(nums[i])        # modify state
        backtrack(i + 1, current)      # i+1 = no reuse
        current.pop()                  # undo
4 — Permutation Pattern N! · used[] · all orderings 
def backtrack(current, used):
    # BASE CASE: all elements placed
    if len(current) == len(nums):
        results.append(current[:])
        return

    # DECISION: pick any unused element
    for i in range(len(nums)):
        if used[i]: continue
        # DEDUP for Permutations II:
        # if i > 0 and nums[i] == nums[i-1] and not used[i-1]: continue

        used[i] = True
        current.append(nums[i])
        backtrack(current, used)
        current.pop()
        used[i] = False
5 — Reusable Choice (Loop Form) i stays · unlimited reuse 
def backtrack(start, rem, current):
    if rem == 0:                          # BASE CASE: exact hit
        results.append(current[:])
        return

    # DECISION: pick element at i (reusable)
    for i in range(start, len(nums)):
        if nums[i] > rem: break               # PRUNE: sorted array

        current.append(nums[i])
        backtrack(i, rem - nums[i], current)  # i not i+1 = allow reuse
        current.pop()
6 — Constraint Backtracking Validate → Modify → Recurse → Undo 
def backtrack(position):
    if all_positions_filled(position):    # BASE CASE: complete solution
        results.append(build_solution())
        return

    # DECISION: loop through all choices for current position
    for choice in get_choices(position):
        if not is_valid(choice, position, constraints):  # VALIDATE first
            continue

        apply(position, choice)              # modify state
        update_constraints(choice)           # modify constraint tracking
        backtrack(next_position(position))   # recurse
        undo_constraints(choice)             # undo constraint tracking
        undo(position, choice)               # undo state
7 — N-Queens O(N!) · 3 constraint sets 
def solveNQueens(n):
    results = []
    cols, diag1, diag2 = set(), set(), set()  # CONSTRAINT TRACKING (state)
    board = [0] * n

    def bt(row):
        if row == n:                           # BASE CASE: all rows filled
            results.append(['.'*c + 'Q' + '.'*(n-c-1) for c in board])
            return

        for col in range(n):                   # DECISION: try each column
            if col in cols or (row-col) in diag1 or (row+col) in diag2:
                continue                       # VALIDATE: skip attacked positions

            board[row] = col               # modify state
            cols.add(col)
            diag1.add(row-col)
            diag2.add(row+col)
            bt(row + 1)                    # TRANSITION: next row
            cols.remove(col)               # undo all state changes
            diag1.remove(row-col)
            diag2.remove(row+col)

    bt(0)
    return results
8 — Sudoku Solver Return bool · find-first solution 
def solveSudoku(board):
    empty = [(r,c) for r in range(9) for c in range(9) if board[r][c] == '.']

    def is_valid(r, c, d):
        if d in board[r]: return False           # check row
        if d in [board[i][c] for i in range(9)]: return False  # col
        br, bc = 3*(r//3), 3*(c//3)
        if d in [board[br+dr][bc+dc] for dr in range(3) for dc in range(3)]:
            return False
        return True

    def bt(idx):
        if idx == len(empty): return True      # BASE CASE: all empty cells filled

        r, c = empty[idx]                    # STATE: current empty cell

        for d in '123456789':                  # DECISION: digits 1-9
            if not is_valid(r, c, d): continue   # VALIDATE

            board[r][c] = d                    # TRANSITION: place digit
            if bt(idx + 1): return True        # recurse to next cell
            board[r][c] = '.'                  # UNDO on failure

        return False                         # BASE CASE (failure): no digit works

    bt(0)

Interview Mindset

Google engineers don't start with code. They start with a state diagram. Your interviewer is watching HOW you think, not just WHAT you produce.

The Pre-Coding Ritual (Do this out loud)

Step-by-Step Protocol — Takes 3-5 minutes, saves 30 minutes of debugging
  1. Restate the goal: "I need to generate/find all X such that Y."
  2. Identify STATE: "At any point in my recursion, I need to know [list state variables]."
  3. Identify DECISIONS: "From any state, I can [list choices]."
  4. Identify TRANSITIONS: "If I make decision X, state changes to Y."
  5. Identify BASE CASES: "I stop when [success condition] or [failure condition]."
  6. Draw a small tree: Show 2-3 levels for a small example. "Let me draw this quickly."
  7. Identify pruning: "I can prune whenever [condition] — here's why that's safe."
  8. Write the template: Now write the code following the template for the identified pattern.

Applied to Subsets

State

"At each call I know which index I'm at and what I've collected so far: (idx, current)."

Decision

"For each element at idx, I decide: include it or exclude it. Two choices always."

Transition

"Both decisions advance idx to idx+1. Include also adds nums[idx] to current."

Base Case

"When idx equals len(nums), every element has been decided. Record current."

Applied to Combination Sum

State

(idx, remaining_sum, current). The index determines which elements are still available. Remaining tells me how much more I need.

Decision

"Two decisions: skip current element (advance idx), or take it (stay at idx since reuse is allowed, subtract from remaining)."

Transition

Skip: idx+1, remaining unchanged. Take: idx stays, remaining - nums[idx].

Base Case

remaining == 0: success. remaining < 0 or idx >= len: failure/return.

Applied to N-Queens

State

(row, cols, diag1, diag2). Row tells me where I'm placing next. The three sets track which positions are attacked.

Decision

"For the current row, I try each column 0 to N-1 — but only those not attacked."

Transition

"Place queen at (row, col). Advance to row+1. Add col, row-col, row+col to constraint sets."

Base Case

"When row equals N, all N queens are placed. This is a valid solution."

Applied to Sudoku

State

(index into empty_cells list, board). Each call decides one empty cell.

Decision

"Digits 1–9. Check validity (row, column, 3×3 box constraint) before placing."

Transition

"Place valid digit d at (r,c). Advance to next empty cell (idx+1)."

Base Case

"Index equals total number of empty cells → board is fully solved."

One-Page Revision Cheat Sheet

Review this in 5 minutes before any interview. It contains everything you need to activate the framework.

The Universal Formula
STATE → DECISION → TRANSITION → BASE CASE

Every recursion/backtracking problem reduces to these four questions. Answer them before writing code.

Pattern Quick-Select
"All subsets/subsequences" → Take/Not Take (binary, 2^n)
"Reuse same element" → Reusable Choice (pass i, not i+1)
"No reuse, duplicates in input" → Loop + sort + dedup
"Order matters, all arrangements" → Permutation (used[])
"Validity constraint at each step" → Constraint Backtracking
"Grid / 2D board" → Grid Backtracking (visited[][])
"Build / cut string" → String Construction (idx into string)
"Overlapping subproblems" → DP Transition (add memoization)
The Index Rule (Critical)
pass i → element at i can be reused (Combination Sum I)
pass i+1 → element at i is done, no reuse (Combination Sum II, Subsets)
no start → permutation mode, all unused elements valid
Deduplication Rules

Combination dedup (Subsets II, Comb Sum II):

Sort first. In loop: if i > start and nums[i] == nums[i-1]: continue

Permutation dedup (Perms II):

Sort first. Skip if nums[i] == nums[i-1] and not used[i-1]

Common Pruning Rules
Sum exceeds target → sort + break when nums[i] > remaining
Too many elements → return if len(current) > k
Constraint violated → skip before modifying state (N-Queens)
No path possible → return False early (Sudoku)
Universal Backtracking Skeleton 
def bt(state):
    if base_case: record/return           # BASE

    for choice in decisions(state):        # DECISION
        if not valid(choice): continue     # PRUNE
        modify(state, choice)              # TRANSITION
        bt(next_state)                     # RECURSE
        undo(state, choice)                # BACKTRACK
5 Mistakes to Actively Avoid
Tracing code mentally instead of drawing the state tree
Forgetting current[:] snapshot — passing reference not copy
Not undoing ALL modifications (esp. constraint tracking structures)
Using i vs i+1 incorrectly in recursive call (reuse vs no-reuse)
Mixing validation and state modification (gatekeeper must not enter the room)
